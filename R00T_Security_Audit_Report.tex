\begin{abstract}
This report presents a comprehensive security audit of the R00T Protocol, a privacy-preserving Automated Market Maker (AMM) leveraging zero-knowledge proofs built on the Ethereum blockchain. The audit covers the core smart contracts including \texttt{ZkAMMv3Router}, \texttt{ZkAMMv3Pair}, \texttt{ZkAMMv3Admin}, \texttt{ZkProjectPool}, \texttt{LaunchpadGovernance}, and supporting infrastructure contracts. We employed static analysis, manual code review, and pattern matching against known vulnerability databases. The protocol demonstrates strong security fundamentals with appropriate use of reentrancy guards, SNARK scalar field validation, and timelocked administrative functions. We identified \textbf{0 High}, \textbf{2 Medium}, and several informational findings. The XMR Bridge components were excluded from this audit scope.
\end{abstract}

\section{Protocol Architecture}
%==============================================================================

\subsection{Overview}

The R00T Protocol implements a privacy-preserving AMM using the UTXO model combined with zero-knowledge proofs. Users interact with the protocol through \textit{commitments} (hiding their balances) and \textit{nullifiers} (preventing double-spending).

\begin{figure}[h]
\centering
\begin{tcolorbox}[width=0.9\textwidth]
\textbf{Core Components:}
\begin{enumerate}[noitemsep]
    \item \textbf{TokenPool}: Sparse Merkle Tree storing commitments
    \item \textbf{ZkAMMv3Pair}: AMM state (ETH/ROOT reserves, LP shares)
    \item \textbf{ZkAMMv3Router}: User-facing operations with proof verification
    \item \textbf{ZkAMMv3Admin}: Timelocked administrative functions
    \item \textbf{ZkProjectPool}: Project token AMM pools
    \item \textbf{NullifierRegistry}: Cross-pool nullifier coordination
\end{enumerate}
\end{tcolorbox}
\caption{Protocol Component Overview}
\end{figure}

\subsection{Commitment Scheme}

The protocol uses Poseidon hashing for commitment generation:

\begin{equation}
\text{commitment} = \text{Poseidon}(\text{amount}, \text{secret}, \text{nullifierKey})
\end{equation}

Nullifiers are derived deterministically:
\begin{equation}
\text{nullifier} = \text{Poseidon}(\text{commitment}, \text{leafIndex}, \text{nullifierKey})
\end{equation}

\subsection{ZK Proof System}

The protocol uses Groth16 proofs on the BN254 curve. Critical security measure: all public inputs are validated against the SNARK scalar field:

\begin{lstlisting}[language=Solidity]
uint256 public constant SNARK_SCALAR_FIELD =
    21888242871839275222246405745257275088548364400416034343698204186575808495617;

// Validation prevents nullifier aliasing attacks
if (nullifierHash >= SNARK_SCALAR_FIELD) revert InvalidScalarField();
\end{lstlisting}

\subsection{AMM Mechanics}

The protocol implements a constant product AMM (xy = k) with:
\begin{itemize}
    \item \textbf{1\% total fee}: 0.3\% protocol fee + 0.7\% LP fee
    \item \textbf{Slippage protection}: \texttt{minAmountOut} parameters
    \item \textbf{Deadline enforcement}: Transaction expiration
\end{itemize}

\section{Security Analysis}
%==============================================================================

\subsection{Access Control}

\subsubsection{Timelock Mechanisms}

Administrative functions implement 48-hour timelocks:

\begin{lstlisting}[language=Solidity]
uint256 public constant ADMIN_TIMELOCK = 48 hours;

function proposeOwnership(address _newOwner) external onlyOwner {
    pendingOwner = _newOwner;
    ownerTimelockExpiry = block.timestamp + ADMIN_TIMELOCK;
    emit OwnershipTransferProposed(owner, _newOwner, ownerTimelockExpiry);
}

function executeOwnershipTransfer() external {
    if (block.timestamp < ownerTimelockExpiry) revert TimelockNotExpired();
    // ... execution
}
\end{lstlisting}

\textbf{Assessment}: \textcolor{lowsev}{SECURE} --- Proper timelock implementation prevents immediate malicious admin actions.

\subsubsection{Emergency Multisig}

Emergency operations require 2-of-3 signatures with 24-hour expiry:

\begin{lstlisting}[language=Solidity]
uint256 public constant EMERGENCY_APPROVAL_EXPIRY = 24 hours;

function _checkAndRecordApproval(bytes32 actionHash) internal returns (bool) {
    // Signature bitmap tracking
    // Approval expiration logic
    if (approvalCount >= 2) {
        // Execute action
        return true;
    }
    return false;
}
\end{lstlisting}

\textbf{Assessment}: \textcolor{lowsev}{SECURE} --- Multi-signature requirement with expiration prevents single-point-of-failure.

\subsection{Reentrancy Protection}

All state-modifying functions use OpenZeppelin's \texttt{ReentrancyGuard}:

\begin{lstlisting}[language=Solidity]
contract ZkAMMv3Router is ReentrancyGuard {
    function sellPrivate(...) external nonReentrant notExpired(deadline) {
        // State changes before external calls
        pair.markNullifierSpent(nullifierHash);
        pair.updateReserves(ethOut, tokenAmount, false);

        // External call last
        pair.sendETH(recipient, ethAfterFees - fee);
    }
}
\end{lstlisting}

\textbf{Assessment}: \textcolor{lowsev}{SECURE} --- Consistent use of reentrancy guards and CEI pattern.

\subsection{Nullifier Security}

\subsubsection{Scalar Field Validation}

All nullifier inputs are validated against the SNARK scalar field to prevent aliasing attacks:

\begin{lstlisting}[language=Solidity]
// Prevents N and N + SNARK_SCALAR_FIELD from being equivalent
if (merkleRoot >= SNARK_SCALAR_FIELD) revert InvalidScalarField();
if (nullifierHash >= SNARK_SCALAR_FIELD) revert InvalidScalarField();
\end{lstlisting}

\textbf{KB Reference}: \texttt{[fv-sol-3-c1-overflow-and-underflow.md]} --- Field arithmetic validation prevents modular aliasing.

\subsubsection{Cross-Pool Coordination}

The \texttt{NullifierRegistry} prevents cross-pool double-spending:

\begin{lstlisting}[language=Solidity]
contract NullifierRegistry {
    mapping(uint256 => bool) public spentNullifiers;
    mapping(address => bool) public authorizedPools;

    function markNullifierSpent(uint256 nullifier) external {
        if (!authorizedPools[msg.sender]) revert Unauthorized();
        if (spentNullifiers[nullifier]) revert NullifierAlreadySpent();
        spentNullifiers[nullifier] = true;
    }
}
\end{lstlisting}

\textbf{Assessment}: \textcolor{lowsev}{SECURE} --- Centralized registry with proper authorization.

\subsection{Front-Running Protection}

\subsubsection{Commitment Binding}

Buy operations use commitment bindings to prevent front-running:

\begin{lstlisting}[language=Solidity]
function buyPrivate(...) external payable nonReentrant {
    bytes32 commitmentBinding = keccak256(
        abi.encodePacked(newCommitment, msg.sender, msg.value)
    );
    pair.useCommitmentBinding(commitmentBinding);
    // ... rest of function
}
\end{lstlisting}

\textbf{KB Reference}: \texttt{[fv-sol-8-c2-front-running.md]} --- Commitment binding prevents sandwich attacks on buys.

\subsubsection{Deadline Enforcement}

Transaction expiration prevents stale transaction execution:

\begin{lstlisting}[language=Solidity]
modifier notExpired(uint256 deadline) {
    if (block.timestamp > deadline) revert TransactionExpired();
    _;
}
\end{lstlisting}

\subsection{MEDIUM-01: Unprotected receive() in ZkAMMv3Router}
%------------------------------------------------------------------------------

\begin{tcolorbox}[colback=orange!10, colframe=orange!80!black, title=Medium Severity]
\textbf{Location}: \texttt{ZkAMMv3Router.sol:881}\\
\textbf{Category}: Fund Management\\
\textbf{Confidence}: 8/10
\end{tcolorbox}

\subsubsection{Description}

The \texttt{ZkAMMv3Router} contract has an unprotected \texttt{receive()} function that accepts ETH from any sender:

\begin{lstlisting}[language=Solidity]
receive() external payable {}
\end{lstlisting}

While the \texttt{ZkAMMv3Pair} correctly restricts its receive function to only accept ETH from the router:

\begin{lstlisting}[language=Solidity]
receive() external payable {
    if (msg.sender != router) revert Unauthorized();
}
\end{lstlisting}

\subsubsection{Impact}

ETH accidentally sent to the Router contract will be permanently stuck with no recovery mechanism. This is lower severity because:
\begin{itemize}
    \item Users typically interact via function calls, not direct ETH transfers
    \item No protocol funds are at risk
    \item Only user error would trigger this
\end{itemize}

\subsubsection{Recommendation}

Add access control or a sweep function:

\begin{lstlisting}[language=Solidity]
// Option 1: Restrict receive
receive() external payable {
    if (msg.sender != address(pair)) revert Unauthorized();
}

// Option 2: Add sweep function
function sweepETH(address payable recipient) external onlyOwner {
    recipient.transfer(address(this).balance);
}
\end{lstlisting}

\subsection{MEDIUM-02: Atomic Swap Commitment Predictability}
%------------------------------------------------------------------------------

\begin{tcolorbox}[colback=orange!10, colframe=orange!80!black, title=Medium Severity]
\textbf{Location}: \texttt{ZkAMMv3Router.sol:833-834}\\
\textbf{Category}: Randomness / MEV\\
\textbf{Confidence}: 8/10
\end{tcolorbox}

\subsubsection{Description}

The atomic swap commitment generation uses sources that validators can influence:

\begin{lstlisting}[language=Solidity]
uint256 r00tCommitment = uint256(keccak256(abi.encodePacked(
    pool,
    blockhash(block.number - 1),  // Validator-controlled
    block.prevrandao,              // Partially predictable
    userEntropy,
    msg.sender,
    r00tAmount,
    projectTokenCommitment,
    currentNonce
))) % SNARK_SCALAR_FIELD;

uint256 r00tNullifier = uint256(keccak256(abi.encodePacked(
    r00tCommitment, "nullifier"
))) % SNARK_SCALAR_FIELD;
\end{lstlisting}

\subsubsection{Impact}

A malicious validator could potentially:
\begin{enumerate}
    \item Predict the commitment value before inclusion
    \item Manipulate \texttt{blockhash} by reordering transactions
    \item Front-run with knowledge of the generated commitment
\end{enumerate}

However, exploitation is limited because:
\begin{itemize}
    \item Requires validator collusion
    \item \texttt{userEntropy} adds unpredictability
    \item Atomic nonce prevents replay
    \item Financial gain is uncertain
\end{itemize}

\subsubsection{Recommendation}

Consider using a commit-reveal scheme or VRF for commitment generation in high-value atomic swaps.

\subsection{INFO-01: LP Share Calculation Race Condition}
%------------------------------------------------------------------------------

\begin{tcolorbox}[colback=blue!10, colframe=blue!80!black, title=Informational]
\textbf{Location}: \texttt{ZkAMMv3Router.sol:589-593}\\
\textbf{Status}: Mitigated by Design
\end{tcolorbox}

The protocol correctly handles LP share race conditions:

\begin{lstlisting}[language=Solidity]
// SECURITY: Only enforce upper bound - user cannot claim MORE than calculated
// No minimum tolerance needed: if user commits to fewer shares due to race
// condition, they only hurt themselves (get less withdrawal + fewer fees)
if (userLpShares > calculatedLpShares) revert InvalidLPShares();
\end{lstlisting}

\textbf{Assessment}: The one-sided check is intentional and secure---users can only under-claim shares, not over-claim.

%------------------------------------------------------------------------------
\subsection{INFO-02: Verifier Lock Is Irreversible}
%------------------------------------------------------------------------------

\begin{tcolorbox}[colback=blue!10, colframe=blue!80!black, title=Informational]
\textbf{Location}: \texttt{ZkAMMv3Admin.sol:151}\\
\textbf{Status}: Design Decision
\end{tcolorbox}

Once \texttt{lockVerifiers()} is called, ZK verifiers cannot be updated:

\begin{lstlisting}[language=Solidity]
function lockVerifiers() external onlyOwner {
    verifiersLocked = true;
    emit VerifiersPermanentlyLocked();
}
\end{lstlisting}

\textbf{Note}: This is a security feature, not a bug. However, if a verifier vulnerability is discovered post-lock, the protocol would need to migrate to new contracts.

%------------------------------------------------------------------------------
\subsection{INFO-03: Fee Epoch Governance Centralization}
%------------------------------------------------------------------------------

\begin{tcolorbox}[colback=blue!10, colframe=blue!80!black, title=Informational]
\textbf{Location}: \texttt{ZkAMMv3Pair.sol:1009-1028}
\end{tcolorbox}

Fee epoch increments are controlled by the owner with mitigations:

\begin{lstlisting}[language=Solidity]
uint256 public constant MIN_EPOCH_DURATION = 7 days;
uint256 public constant MIN_CLAIM_WINDOW = 72 hours;

function announceEpochIncrement() external onlyRouter {
    if (block.timestamp < lastEpochIncrementTime + MIN_EPOCH_DURATION)
        revert EpochTooSoon();
    if (block.timestamp < lastLPDepositTime + MIN_CLAIM_WINDOW)
        revert RecentLPDeposit();
    // ...
}
\end{lstlisting}

\textbf{Assessment}: Acceptable with existing safeguards (7-day minimum, 72-hour claim window).

%------------------------------------------------------------------------------
\subsection{INFO-04: Emergency Insert Authorization Window}
%------------------------------------------------------------------------------

\begin{tcolorbox}[colback=blue!10, colframe=blue!80!black, title=Informational]
\textbf{Location}: \texttt{TokenPool.sol:381-390}
\end{tcolorbox}

The emergency insert mechanism has a 30-day delay after authorization revocation:

\begin{lstlisting}[language=Solidity]
uint256 public constant EMERGENCY_INSERT_DELAY = 30 days;

function emergencyInsert(uint256 leaf) external returns (uint256) {
    if (!wasEverAuthorized[msg.sender]) revert NeverAuthorizedInR00TPool();
    if (block.timestamp < authorizationRevokedAt[msg.sender] + EMERGENCY_INSERT_DELAY)
        revert EmergencyDelayNotMet();
    // ...
}
\end{lstlisting}

\textbf{Assessment}: Properly designed for claim recovery scenarios.

\section{Recommendations}
%==============================================================================

\begin{enumerate}
    \item \textbf{Router receive() Protection}: Add access control to prevent ETH getting stuck
    \item \textbf{Atomic Swap Entropy}: Consider VRF or commit-reveal for high-value swaps
    \item \textbf{Circuit Monitoring}: Implement monitoring for unusual proof verification patterns
    \item \textbf{Emergency Procedures}: Document and test emergency withdrawal procedures
    \item \textbf{Upgrade Path}: Plan migration strategy if post-lock verifier issues arise
\end{enumerate}

%==============================================================================
\section{Conclusion}
%==============================================================================

The R00T Protocol demonstrates mature security practices for a privacy-preserving AMM:

\begin{itemize}
    \item \textbf{Strong fundamentals}: Proper reentrancy guards, timelocks, and access control
    \item \textbf{ZK-specific security}: SNARK scalar field validation prevents nullifier aliasing
    \item \textbf{DeFi best practices}: Slippage protection, minimum liquidity, LP lock periods
    \item \textbf{Defense in depth}: Multiple layers of protection for critical operations
\end{itemize}

The two medium-severity findings are edge cases with limited practical impact. We recommend addressing them in a future update while acknowledging they do not pose immediate risk to user funds.
