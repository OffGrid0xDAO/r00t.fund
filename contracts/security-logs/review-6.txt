# Security Audit Report: HideMyCoin Smart Contracts

After thorough analysis of the Solidity smart contracts in this branch, I've identified the following **HIGH-CONFIDENCE vulnerabilities**:

---

## Vuln 1: `ZkAMMPair.sol:575-593`

* **Severity**: High  
* **Category**: fund_mismanagement
* **KB Reference**: [fv-sol-5-c3-improper-state-transitions.md] - Virtual reserve pattern without actual token backing
* **Description**: The `withdrawPublic()` function in ZkAMMPair decrements `tokenReserve` but never actually transfers any ERC20 tokens. The contract only tracks virtual reserves without holding real tokens. Users burning their commitments to "withdraw" tokens receive nothing - just an event emission.
* **Exploit Scenario**: A user with a valid project token commitment in ZkAMMPair calls `withdrawPublic()`, passes proof verification, the nullifier is spent and `tokenReserve` is decremented, but no actual tokens are transferred to the recipient. The user loses their commitment permanently with no value received.
* **Recommendation**: Either:
  1. Remove `withdrawPublic()` from ZkAMMPair since it operates on virtual reserves only, OR
  2. Add actual ERC20 token holding and transfer logic like ZkProjectPool does: `token.safeTransfer(recipient, amount)`
* **Confidence**: 9/10

---

## Vuln 2: `LaunchpadGovernance.sol:989-1013`

* **Severity**: Medium  
* **Category**: logic_error, fund_mismanagement
* **KB Reference**: [fv-sol-4-c2-unrestricted-role-assignment.md] - Privilege escalation allowing owner to bypass claim process
* **Description**: In `withdrawFees()`, the owner creates a claim with `creator: platformTreasury`, but `setClaimCommitment()` requires `msg.sender == claim.creator`. If `platformTreasury` is a different address than `owner`, the owner cannot set the commitment for fee claims they created. The owner must use `setTreasuryClaimCommitment()` but this only works if `claim.creator == platformTreasury`.
* **Exploit Scenario**: If owner != platformTreasury address: (1) Owner calls `withdrawFees()` which creates a claim with `creator = platformTreasury`, (2) Owner cannot call `setClaimCommitment()` as they are not the creator, (3) `platformTreasury` address may be a multi-sig or contract that cannot interact properly with this function, (4) After 30 days, anyone can process via emergency but the commitment may never be set.
* **Recommendation**: Either ensure owner can always set commitment for treasury claims, or require commitment to be provided at the time of `withdrawFees()` call with validation.
* **Confidence**: 8/10

---

## Vuln 3: `ZkProjectPool.sol:982-1040` 

* **Severity**: Medium  
* **Category**: access_control
* **KB Reference**: [fv-sol-4-c2-unrestricted-role-assignment.md] - Missing validation on output commitment in claimDevAllocation
* **Description**: The `claimDevAllocation()` function does not validate that the `commitment` parameter is within the SNARK scalar field. Unlike all other functions in the contract that accept commitments (which validate `if (commitment >= SNARK_SCALAR_FIELD) revert InvalidScalarField()`), this function accepts any uint256 value for `commitment`.
* **Exploit Scenario**: Creator calls `claimDevAllocation()` with a commitment >= SNARK_SCALAR_FIELD. The commitment is inserted into the merkle tree. When the creator later tries to spend this commitment via a ZK proof, the proof will fail because the circuit operates modulo SNARK_SCALAR_FIELD, meaning the commitment cannot be proven to exist. The creator permanently loses their dev allocation tokens.
* **Recommendation**: Add validation at the start of `claimDevAllocation()`:
  ```solidity
  if (commitment >= SNARK_SCALAR_FIELD) revert InvalidScalarField();
  ```
* **Confidence**: 9/10

---

## Vuln 4: `ZkProjectPool.sol:957-980`

* **Severity**: Medium  
* **Category**: logic_error
* **KB Reference**: [fv-sol-5-c3-improper-state-transitions.md] - Missing SNARK scalar field validation
* **Description**: The `depositPublic()` function does not validate that the `commitment` parameter is within the SNARK scalar field. While other deposit/swap functions properly check `if (commitment >= SNARK_SCALAR_FIELD) revert InvalidScalarField()`, this function only checks `if (commitment == 0)`.
* **Exploit Scenario**: A user calls `depositPublic()` with a valid `depositorBinding` but a commitment value >= SNARK_SCALAR_FIELD. The ERC20 tokens are transferred from the user to the pool, and the commitment is inserted into the merkle tree. However, when the user later tries to spend this commitment via a ZK proof, the proof will fail because the circuit computes modulo SNARK_SCALAR_FIELD. The user permanently loses their deposited tokens.
* **Recommendation**: Add validation:
  ```solidity
  if (commitment >= SNARK_SCALAR_FIELD) revert InvalidScalarField();
  ```
* **Confidence**: 9/10

---

## Vuln 5: `ZkAMMv3.sol:490-543` 

* **Severity**: Medium  
* **Category**: logic_error
* **KB Reference**: [fv-sol-5-c3-improper-state-transitions.md] - Missing commitment scalar field validation
* **Description**: The `buyPrivate()` function validates `newCommitment >= SNARK_SCALAR_FIELD` but allows commitments of exactly 0 to pass through (there's no `if (newCommitment == 0) revert ZeroAmount()` check). A zero commitment is invalid and cannot be spent.
* **Exploit Scenario**: User sends ETH calling `buyPrivate()` with `newCommitment = 0`. The transaction succeeds: ETH is added to reserves, tokens are removed from reserves, fees are accumulated, and commitment 0 is inserted into the merkle tree. However, commitment 0 cannot be meaningfully spent via ZK proofs. The user loses their ETH permanently.
* **Recommendation**: Add check for zero commitment:
  ```solidity
  if (newCommitment == 0) revert ZeroAmount();
  ```
* **Confidence**: 9/10

---

## Vuln 6: `ZkAMMv3.sol:985-1027`

* **Severity**: Medium  
* **Category**: logic_error
* **KB Reference**: [fv-sol-5-c3-improper-state-transitions.md] - Missing commitment validation in bootstrapLiquidity
* **Description**: The `bootstrapLiquidity()` function only checks `if (lpCommitment == 0)` but does NOT check if `lpCommitment >= SNARK_SCALAR_FIELD`. An LP commitment outside the scalar field cannot be spent via ZK proofs for liquidity removal or fee claims.
* **Exploit Scenario**: Owner calls `bootstrapLiquidity()` with a valid ETH amount but an `lpCommitment` >= SNARK_SCALAR_FIELD. The LP position is created with shares tracked, but the owner can never remove their liquidity or claim fees because ZK proofs require the commitment to be within the scalar field.
* **Recommendation**: Add validation:
  ```solidity
  if (lpCommitment >= SNARK_SCALAR_FIELD) revert InvalidProof();
  ```
* **Confidence**: 9/10

The most critical issue is **Vuln 1** in ZkAMMPair.sol where users can permanently lose their commitments by calling `withdrawPublic()` on a contract that has no actual tokens to transfer.
