# Security Review Report: HideMyCoin/contracts

## Summary

The codebase shows evidence of extensive security work with many pre-existing security fixes. The contracts implement complex ZK-SNARK-based privacy-preserving AMM functionality with governance, launchpad, and fee distribution features. After careful review, I found **1 HIGH** and **2 MEDIUM** severity issues with high confidence.

---

# Vuln 1: `ZkProjectPoolImpl.sol:306`

* **Severity**: High  
* **Category**: fund_mismanagement, reentrancy  
* **KB Reference**: [fv-sol-1-c3-cross-contract.md] - Similar cross-contract reentrancy via external call before state completion
* **Description**: In `atomicSwapFromR00T`, the `nullifierRegistry.markSpent()` external call is made BEFORE all state updates are complete. Specifically, fee distribution (`feePerShare` update and `accumulatedLPFees` increment) happens AFTER the external call at lines 293-298 while `markSpent()` is called at line 306. If `nullifierRegistry` is compromised or allows reentrant callbacks, an attacker could exploit stale `feePerShare` values during fee claims.

```solidity
// INTERACTIONS: External call to nullifier registry AFTER all state updates
// SECURITY FIX (Audit Vuln 1): Moved markSpent() after state updates to follow CEI pattern
// This prevents cross-contract reentrancy if nullifierRegistry is compromised
nullifierRegistry.markSpent(r00tNullifier);  // Line 306
```

However, the reserve updates at lines 288-290 ARE complete before this call. The vulnerability is that fee accounting (`feePerShare += ...` at line 294) happens BEFORE the external call, which is correct. 

**Re-analysis**: Looking more carefully, the state updates (lines 287-298) all happen BEFORE `nullifierRegistry.markSpent()` (line 306). The CEI pattern appears correctly applied here.

* **Confidence**: 6/10 (Below threshold - likely false positive after re-analysis)

# Vuln 2: `ZkProjectPool.sol:1400-1401`

* **Severity**: High  
* **Category**: reentrancy, fund_mismanagement  
* **KB Reference**: [fv-sol-1-c3-cross-contract.md] - Cross-contract reentrancy pattern where external call precedes state updates
* **Description**: In `atomicSwapFromR00T()` (lines 1384-1434), the function calls `nullifierRegistry.markSpent()` at line 1401 **BEFORE** critical state updates (reserve updates at lines 1413-1425). This violates the checks-effects-interactions (CEI) pattern. If the `nullifierRegistry` contract is compromised or contains a callback mechanism, an attacker could re-enter during `markSpent()` while reserves haven't been updated, potentially draining funds through price manipulation.

* **Exploit Scenario**: 
  1. Attacker triggers `atomicSwapFromR00T()` via ZkAMMv3
  2. During `nullifierRegistry.markSpent()` call (line 1401), malicious registry calls back into the contract
  3. Re-entrant call sees old `r00tReserve` and `tokenReserve` values
  4. Attacker manipulates pricing calculations or executes additional swaps at favorable rates

* **Recommendation**: Move `nullifierRegistry.markSpent(r00tNullifier)` to AFTER all state updates (after line 1425). The correct pattern should be:
  ```solidity
  // 1. CHECKS (done)
  // 2. EFFECTS - all state updates FIRST
  r00tReserve += r00tAfterFees;
  tokenReserve -= tokensOut;
  // ... all other state updates ...
  
  // 3. INTERACTIONS - external calls LAST
  nullifierRegistry.markSpent(r00tNullifier);
  uint256 leafIndex = projectTokenPool.insert(outputCommitment);
  ```

* **Confidence**: 9/10

# Vuln 3: `LaunchpadGovernanceV2.sol:434-443`

* **Severity**: Medium  
* **Category**: logic_error, fund_mismanagement  
* **KB Reference**: [fv-sol-5-c3-improper-state-transitions.md] - Improper state validation allowing unintended operations
* **Description**: In `setClaimCommitment()`, there's no validation that `outputCommitment` is within the SNARK scalar field. The function allows setting any `uint256` value as the commitment. If a user sets `outputCommitment >= SNARK_SCALAR_FIELD`, the commitment becomes unspendable via ZK proofs (due to field overflow), permanently locking their funds.

```solidity
function setClaimCommitment(uint256 claimId, uint256 outputCommitment, bytes calldata encryptedNote) external {
    PendingR00tClaim storage claim = pendingR00tClaims[claimId];
    if (claim.amount == 0) revert InvalidClaimId();
    if (claim.claimed) revert ClaimAlreadyProcessed();
    if (msg.sender != claim.creator) revert NotClaimCreator();
    if (outputCommitment == 0) revert ZeroAddress();
    // MISSING: if (outputCommitment >= SNARK_SCALAR_FIELD) revert InvalidScalarField();
    
    claim.outputCommitment = outputCommitment;
    // ...
}
```

* **Exploit Scenario**: 
  1. User creates a rejected proposal and calls `withdrawRejected()` to get a claim
  2. User calls `setClaimCommitment()` with `outputCommitment = SNARK_SCALAR_FIELD` (accidentally or maliciously)
  3. Owner processes the claim, creating a commitment in r00tPool
  4. The commitment is permanently unspendable - user's R00T refund is locked forever

* **Recommendation**: Add validation:
  ```solidity
  if (outputCommitment >= SNARK_SCALAR_FIELD) revert InvalidScalarField();
  ```

* **Confidence**: 9/10

# Vuln 4: `LaunchpadGovernance.sol:711-722`

* **Severity**: Medium  
* **Category**: logic_error, fund_mismanagement  
* **KB Reference**: [fv-sol-5-c3-improper-state-transitions.md] - Missing input validation allowing unspendable state
* **Description**: Same issue as Vuln 3 - `setClaimCommitment()` in LaunchpadGovernance lacks SNARK scalar field validation for `outputCommitment`. Users can set commitments >= SNARK_SCALAR_FIELD, making their refunds permanently unspendable.

* **Exploit Scenario**: Same as Vuln 3 - user loses refund permanently.

* **Recommendation**: Add the same validation as the other functions that handle commitments:
  ```solidity
  if (outputCommitment >= SNARK_SCALAR_FIELD) revert InvalidScalarField();
  ```

* **Confidence**: 9/10

## False Positive Analysis

Several patterns that initially appeared concerning were verified as safe:

1. **ZkAMMv3 collectProtocolFees()** - Uses CEI pattern correctly with state update before external call
2. **ZkProjectPool swapR00tForToken()** - State updates happen before `nullifierRegistry.markSpent()` (lines 472-492 before line 492)
3. **Emergency functions** - 30-day delays provide adequate protection
4. **Epoch-based fee claiming** - Properly designed to prevent double-claiming across epochs

## Notes on Existing Security Fixes

The codebase contains many well-implemented security fixes including:
- Timelocked admin functions (48-hour delays)
- SNARK scalar field validation in most places
- ReentrancyGuard on critical functions
- Slippage protection
- LP lock periods
- Emergency recovery mechanisms
- 2-of-3 multisig for emergency actions
