# Security Audit Report: HideMyCoin Contracts

## Executive Summary

The contracts demonstrate substantial security awareness with extensive use of ReentrancyGuard, CEI pattern, admin timelocks, SNARK scalar field validation, and vesting mechanisms. Most previously identified vulnerabilities appear to have been addressed. However, I identified the following issues that warrant attention:

---

# Vuln 1: `ZkAMMv3.sol:486-487` - Commitment Front-Running in buyPrivate

* **Severity**: Medium  
* **Category**: front-running  
* **KB Reference**: [fv-sol-8-c2-front-running.md] - Front-running vulnerability pattern where transactions can be observed and exploited
* **Description**: In `buyPrivate()`, users provide their own `newCommitment` value as a parameter. An attacker monitoring the mempool can see a pending transaction with a specific commitment and front-run it with the same commitment value. Since the same commitment can be inserted into the Merkle tree, the original user's funds would create a duplicate commitment, but they wouldn't be able to generate a valid ZK proof for it (as they don't know the secrets the attacker used).
* **Exploit Scenario**: 
  1. User A submits a `buyPrivate` transaction with commitment `C` (derived from their secret)
  2. Attacker sees this in mempool and front-runs with higher gas, inserting commitment `C` into the tree
  3. User A's transaction succeeds but now commitment `C` exists twice
  4. Only the attacker knows the secret for the first insertion - User A cannot generate a valid proof for the original commitment
* **Recommendation**: Add a `depositorBinding` parameter similar to `depositPublic()` in ZkProjectPool:711-954, which computes `expectedBinding = keccak256(abi.encodePacked(commitment, msg.sender, amount))` to bind commitments to specific depositors.
* **Confidence**: 8/10

# Vuln 2: `ZkProjectPool.sol:721-722` - LP Share Rounding Validation Only on Subsequent Deposits

* **Severity**: Medium  
* **Category**: precision_errors  
* **KB Reference**: [fv-sol-2-c3-rounding.md] - Rounding errors in share calculations
* **Description**: The stricter rounding check (`actualProduct * 10000 >= expectedProduct * 9995`) at line 731 is only applied when `totalLPShares > 0`. The first LP deposit (when `totalLPShares == 0`) uses `sqrt(r00tAmount * tokenAmount)` without rounding validation. The `MINIMUM_LIQUIDITY` burn at pool construction (line 392) helps mitigate this, but the initial LP calculation path in `addLiquidity()` lacks explicit rounding protection.
* **Exploit Scenario**: While mitigated by `MINIMUM_LIQUIDITY` burn, edge cases with carefully chosen initial values could potentially inflate shares disproportionately on the first LP add.
* **Recommendation**: Add explicit validation for the first LP deposit path in `addLiquidity()` or document that the `MINIMUM_LIQUIDITY` from construction fully prevents this attack vector.
* **Confidence**: 8/10

---

# Vuln 3: `NullifierRegistry.sol:109-112` - Pool Authorization Cooldown Bypass via Batch Authorization

* **Severity**: Medium  
* **Category**: logic_errors  
* **KB Reference**: [fv-sol-5-c3-improper-state-transitions.md] - Logic allowing operations to bypass intended safeguards
* **Description**: The `POOL_AUTHORIZATION_COOLDOWN` of 24 hours is enforced in `onlyAuthorizedPool` modifier for `markSpent()` and `checkAndMark()`. However, if a compromised governance calls `setPoolAuthorization(pool, true)` and then immediately calls `setPoolAuthorization(pool, false)` followed by `setPoolAuthorization(pool, true)`, the `poolAuthorizationTime[pool]` gets reset to the current timestamp. This resets the 24-hour cooldown clock. If combined with block reordering on L2s during the original cooldown period, an attacker could exploit timing windows.
* **Exploit Scenario**: 
  1. Compromised governance authorizes malicious pool at T=0
  2. After 23 hours (before cooldown expires), governance deauthorizes then immediately re-authorizes
  3. Cooldown resets to T=23h, requiring only 1 more hour
  4. This allows reducing effective cooldown through strategic toggling
* **Recommendation**: Track original authorization time separately from latest authorization time, or prevent re-authorization within `POOL_AUTHORIZATION_COOLDOWN` of a previous deauthorization.
* **Confidence**: 8/10

# Vuln 4: `LaunchpadGovernance.sol:1021-1036` - Missing SNARK Scalar Field Validation in setTreasuryClaimCommitment

* **Severity**: Medium  
* **Category**: logic_errors  
* **KB Reference**: [fv-sol-5-c2-incorrect-conditionals.md] - Missing validation for critical inputs
* **Description**: The function `setTreasuryClaimCommitment()` at line 1021 allows the owner to set a commitment for treasury fee claims. However, unlike `setClaimCommitment()` at line 724 which validates `if (outputCommitment >= SNARK_SCALAR_FIELD) revert InvalidScalarField()`, the treasury version does not validate the commitment against `SNARK_SCALAR_FIELD`. This could allow setting a commitment that cannot be spent via ZK proofs.
* **Exploit Scenario**: Owner (or compromised owner) sets a treasury claim commitment >= `SNARK_SCALAR_FIELD`. When processed, the commitment is inserted into the Merkle tree, but it can never be spent because ZK circuits cannot generate valid proofs for values outside the scalar field. This permanently locks those R00T funds.
* **Recommendation**: Add `if (outputCommitment >= SNARK_SCALAR_FIELD) revert InvalidScalarField();` to `setTreasuryClaimCommitment()`.
* **Confidence**: 9/10

---

# Vuln 5: `ZkAMMv3.sol:1202-1248` - Epoch Increment Announcement Cannot Be Cancelled By LPs

* **Severity**: Medium  
* **Category**: access_control  
* **KB Reference**: [fv-sol-4-c3-lack-of-multi-signature-for-crucial-operations.md] - Critical operations lacking adequate oversight
* **Description**: While the 24-hour claim window before epoch increment is a good protection, only the owner can cancel the epoch increment via `cancelEpochIncrement()`. If the owner announces an epoch increment during a period of low LP activity (holidays, weekends), and LPs cannot claim in time, they lose their accumulated fees for that epoch. There's no mechanism for LPs to delay or contest the epoch increment.
* **Exploit Scenario**: 
  1. Owner announces epoch increment on Friday evening
  2. Many LPs are unavailable over the weekend
  3. After 24 hours, owner executes increment, resetting feePerShareAtEpochStart
  4. LPs who didn't claim lose their accumulated fees from the previous epoch
* **Recommendation**: Consider extending MIN_CLAIM_WINDOW to 72 hours, or allow a quorum of LPs (via ZK proof of LP ownership) to delay epoch increments by an additional period.
* **Confidence**: 8/10
